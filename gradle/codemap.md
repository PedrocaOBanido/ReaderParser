# gradle/

## Responsibility

Centralised build configuration for the single-module Android project. Contains
three files that together establish every dependency version, plugin reference,
JVM toolchain, and Gradle wrapper binary used by the project:

- **`libs.versions.toml`** — Gradle version catalog (declared in `build.gradle.kts`
  via `enableFeaturePreview("VERSION_CATALOGS")` or the default catalog block).
  Single source of truth for all library coordinates, plugin coordinates, and
  their version numbers.
- **`gradle-daemon-jvm.properties`** — Auto-generated JDK toolchain URLs for
  the Gradle daemon JVM (target JDK 21, sourced from foojay.io disco API).
- **`wrapper/`** — Contains `gradle-wrapper.jar` (the bootstrap binary) and
  `gradle-wrapper.properties` (the distribution URL and checksum). Allows
  reproducible builds without a pre-installed Gradle runtime.

There is no multi-module build structure; a future split into `:core`, `:domain`,
`:data`, `:sources:<site>`, `:feature:<screen>`, `:app` would add per-module
`build.gradle.kts` files but the version catalog would remain shared.

## Design

### Version catalog (`libs.versions.toml`)

The file follows the TOML-based catalog format (three sections):

| Section | Purpose | Example |
|---|---|---|
| `[versions]` | Declare version strings once; reference them in `[libraries]` and `[plugins]` via `version.ref` | `kotlin = "2.1.20"` |
| `[libraries]` | Map human-readable names to Maven coordinates `(group:artifact:version)` | `compose-material3 = { module = "androidx.compose.material3:material3" }` — version inherited from BOM |
| `[plugins]` | Declare Gradle plugin references that `plugins { alias(...) }` blocks consume in `build.gradle.kts` | `kotlin-compose = { id = "org.jetbrains.kotlin.plugin.compose", version.ref = "kotlin" }` |

Key design decisions reflected in the catalog:

1. **BOM-driven Compose.** `compose-bom` is the single version source for all
   `androidx.compose.*` libraries. Individual Compose entries (`compose-ui`,
   `compose-material3`, etc.) omit `version.ref` and inherit from the BOM.
2. **Single Ktor version.** All Ktor modules (`client-core`, `client-okhttp`,
   `client-content-negotiation`, `serialization-json`, `client-cio`, `client-mock`)
   share `ktor = "3.0.1"`.
3. **Room with KSP.** Room uses `room-compiler` via KSP (`ksp(...)` in build.gradle.kts),
   not `kapt`. The catalog entry exists to pin the compiler version to the runtime.
4. **Testing libraries** are enumerated alongside production deps: `turbine`,
   `truth`, `kotlinx-coroutines-test`, `room-testing`, `work-testing`,
   `coil-test`, `ktor-client-cio`/`ktor-client-mock`, `compose-ui-test-junit4`.
   This avoids version drift between test and production classpaths.
5. **Hilt + WorkManager integration.** `hilt-work` provides `HiltWorkerFactory`
   and `hilt-navigation-compose` provides `hiltViewModel()` in composable nav.

### Daemon JVM properties (`gradle-daemon-jvm.properties`)

Generated by the `updateDaemonJvm` Gradle plugin (not visible in build.gradle.kts
but present in the project's plugin classpath). Maps OS + architecture combinations
to JDK 21 download URLs from foojay.io. This ensures the Gradle daemon runs on
a consistent JDK version regardless of the host machine's default JVM, preventing
toolchain mismatches.

### Wrapper (`wrapper/`)

Standard Gradle wrapper. `gradle-wrapper.properties` pins the distribution to a
specific Gradle version (e.g., `gradle-8.x-bin.zip`). The JAR is the bootstrap
loader — checked into VCS so `./gradlew` works without a pre-installed Gradle.

## Flow

```
Developer / CI
  → ./gradlew <task>       // invokes gradle-wrapper.jar
    → wrapper downloads + runs pinned Gradle distribution
      → Gradle daemon starts on JDK 21 (from daemon-jvm.properties)
        → settings.gradle.kts loads project
          → libs.versions.toml is parsed into version catalog
            → build.gradle.kts resolves dependencies via catalog aliases
              → AGP + KSP + Kotlin Compiler compile the app
```

Dependency resolution:

```
libs.versions.toml:
  [versions] kotlin = "2.1.20"
  [libraries] ktor-client-core = { module = "io.ktor:ktor-client-core", version.ref = "ktor" }

build.gradle.kts:
  dependencies { implementation(libs.ktor.client.core) }
    → resolves io.ktor:ktor-client-core:3.0.1
```

## Integration

### Connected files

| File | What references it |
|---|---|
| `libs.versions.toml` | `build.gradle.kts` via `alias(libs.plugins.xxx)` and `implementation(libs.xxx.yyy)` |
| `gradle-daemon-jvm.properties` | Implicitly read by the `updateDaemonJvm` plugin at configuration time |
| `wrapper/gradle-wrapper.properties` | Read by `gradle-wrapper.jar` on every `./gradlew` invocation |

### Module relationships

The project currently has a single `:app` module whose `build.gradle.kts` applies
plugins and consumes catalog entries directly. If the project is later split into
multiple modules, each module's `build.gradle.kts` imports the shared catalog
via `libs.*` references — no catalog duplication needed.

### CI integration

The `scripts/ci-check` and `scripts/pre-push` scripts invoke `./gradlew` tasks
(`:app:lintDebug`, `:app:testDebugUnitTest`, `:app:connectedDebugAndroidTest`,
`:app:assembleDebug`) which resolve dependencies against this catalog. The catalog
is therefore a dependency of the CI pipeline: any version bump or new library
addition in `libs.versions.toml` directly affects what CI compiles and tests.

### Adding a dependency

1. Add version to `[versions]` (only if not already present from another lib).
2. Add library entry in `[libraries]` referencing the version.
3. Optionally add plugin entry in `[plugins]`.
4. Consume in `build.gradle.kts` via `libs.plugins.xxx` / `libs.xxx.yyy`.
5. Sync Gradle and verify with `./gradlew :app:assembleDebug`.
